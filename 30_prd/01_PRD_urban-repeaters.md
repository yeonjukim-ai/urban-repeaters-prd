본 PRD는 상기 VPS 및 분석 문서를 기반으로, MVP(1차 릴리즈) 범위와 성공 기준을 제품 개발 관점에서 구체화한 문서이다.

1. 개요
1.1 제품 개요

도심형 통학 관리형 재수 학원을 위한 성과 리포트 자동화 SaaS 서비스로,
주 10~20시간 소요되던 리포트 제작을 자동화하고, 여러 시스템에 분산된 데이터를 단일 대시보드로 통합하여 실시간 모니터링과 분석을 제공한다.

1.2 제품 목표

리포트 제작 시간을 주 15시간에서 주 1시간 이하로 단축 (93% 이상 절감)

학부모 상담 시 30초 이내 종합 성과 리포트 생성

정기 리포트 자동 전송으로 학부모 만족도 및 재등록률 향상

데이터 기반 의사결정 지원을 통한 학원 운영 효율화

1.3 타깃 시장

상위 레벨(TAM/SAM)에서는 입시 관련 학원 전체를 고려하되,

본 제품(MVP)과 PRD에서는 다음과 같이 도심형 통학 관리형 재수 학원을 핵심 타깃으로 한정한다.

1차 타깃 (핵심)

중형 도심형 통학 관리형 학원

지점 수: 2~5개

학생 수: 200~500명

전국 기준 약 300~500개소 중, 초기 3년 내 30~50개소 확보를 목표

2차 타깃

소형 단일 지점 도심형 학원 (1개 지점, 학생 수 50~150명)

3차 타깃

프리미엄 소규모 학원 (30명 이하, 고액 수강료, 학부모 커뮤니케이션 전략적 비중이 높음)

2. 문제 정의
2.1 핵심 문제

도심형 통학 관리형 재수 학원들은 학부모 커뮤니케이션과 데이터 기반 의사결정을 위해 정기적으로 성과 리포트를 제작해야 하지만, 현재는 수동 작업에 의존하여 과도한 시간과 인력을 소비하고 있다.

2.2 구체적 Pain Point
2.2.1 리포트 제작 시간 과다 소비

현황: 주 10~20시간을 리포트 제작에 소비 (학원당 연간 약 700시간)

원인:

여러 시스템(출석 앱, LMS, 모의고사 플랫폼, 결제 시스템)을 오가며 데이터 수집

엑셀 템플릿 복사 및 수동 데이터 입력

학생별 리포트를 일일이 제작

영향: 전략적 업무에 집중할 시간 부족, 인건비 부담 증가

2.2.2 데이터 소스 분산 및 수동 통합

현황: 4~5개 이상의 시스템에서 데이터를 수동으로 취합

원인:

각 시스템이 독립적으로 운영됨

API 연동이 없거나 어려움

데이터 형식이 상이함

영향:

데이터 통합 시간 주 5시간 소요

시스템 간 데이터 불일치 발생 (정확도 85% 수준)

실수로 인한 신뢰도 저하

2.2.3 학부모 상담 준비 시간 부족

현황: 상담 1~2시간 전에 급하게 리포트 제작, 상담당 평균 1시간 소요

원인:

학생별 리포트를 상담 직전에 수동 제작

여러 시스템을 오가며 최신 데이터 확인

영향: 리포트 품질 저하, 상담 효율성 감소, 학부모 신뢰도 하락

2.2.4 리포트 일관성 및 전문성 부족

현황: 리포트 형식이 매번 달라지고, 전문성이 떨어져 보임

원인:

수동 제작으로 인한 형식 불일치

디자인 및 레이아웃 표준화 부재

영향: 학원 브랜드 이미지 저하, 경쟁 학원 대비 차별화 부족

2.2.5 문제 학생 조기 발견의 어려움

현황: 학생별 데이터를 수동으로 확인하여 문제 발견이 지연됨

원인:

학생 수가 많아 일일이 확인하기 어려움

여러 지표를 종합적으로 분석하기 어려움

영향: 학생 이탈률 증가, 재등록률 저하

2.3 현재 대체 솔루션의 한계

엑셀/구글 시트: 수동 작업, 시간 소비, 실수 빈발

대형 AMS: 과도한 기능, 높은 가격 (월 수백만원), 중소 학원에 부적합

범용 BI 도구: 교육 도메인 특화 기능 부족, 학습 곡선 높음

외부 아웃소싱: 비용 부담 (월 수십만원~수백만원), 시간 지연, 실시간 대응 어려움

3. 목적 및 성공 기준
3.1 제품 목적

리포트 제작 시간을 93% 이상 절감하여 전략적 업무에 집중할 시간 확보

여러 시스템 데이터를 자동 통합하여 데이터 정확성 99% 이상 달성

학부모 상담 시 30초 이내 종합 리포트 생성으로 전문성 향상

정기 리포트 자동 전송으로 학부모 만족도(NPS) 50 → 65 이상 향상

문제 학생 조기 발견으로 재등록률 70% → 75% 이상 향상

3.2 성공 기준 (90일 MVP 기준)
3.2.1 제품 개발 성공 기준

MVP 기능(F1~F4) 개발 완료율 100%

주요 3개 시스템(출석 앱, LMS, 모의고사 플랫폼) API/파일 연동 완료율 100%

학생별 리포트 생성 속도 30초 이내 달성률 95% 이상

데이터 통합 정확도 99% 이상

시스템 가동률 99% 이상

3.2.2 고객 확보 성공 기준

파일럿 고객 3~5개 학원 확보

파일럿 고객 만족도(NPS) ≥ 50

리포트 사용 빈도 주 2회 이상 (사용자당)

MVP 기능(F1~F4) 사용률 80% 이상

고객 지원 응답 시간 24시간 이내

3.2.3 비즈니스 임팩트 성공 기준

리포트 제작 시간: 주 15시간 → 주 1시간 이하 (93% 절감)

데이터 통합 시간: 주 5시간 → 주 30분 이하 (90% 절감)

리포트 정확도: 85% → 99% 이상

학부모 만족도(NPS): 50 → 65 이상

파일럿 고객 중 80% 이상 정식 구독 전환

월 MRR 500만원 달성 (파일럿 3개 전환 + 신규 2개)

상세 KPI 표와 측정 계획은 8장에서 재정의(Tracking 용)하며, 3장은 MVP의 상위 성공 정의로 사용한다.

4. 타깃 사용자 & 페르소나 요약
4.1 핵심 페르소나
페르소나 1: 김원장 (45세, 원장)

역할: 도심형 통학 관리형 재수 학원 원장 (지점 3개, 학생 수 약 400명)

목표: 재등록률 70%→85%, 학부모 NPS 50→70, 인건비 10% 절감

KPI: 월 매출, 순이익률, 재등록률, 학부모 NPS, 학생당 평균 수강 기간

핵심 Pain: 리포트 제작에 주 5~6시간 소비, 여러 시스템 데이터 수동 통합, 전략적 업무 집중 시간 부족

디지털 리터러시: 중상 (엑셀, 구글 시트 기본 사용 가능, 클릭 기반 메뉴 방식 도구 선호, 복잡한 명령어 입력 불필요)

페르소나 2: 박학사 (32세, 학사 관리 담당자)

역할: 학사 관리 팀장 (학생 수 약 300명)

목표: 리포트 제작 시간 주 15시간→주 3시간 이하, 학생별 맞춤 관리 케이스 조기 발견

KPI: 리포트 제작 소요 시간, 학부모 상담 만족도, 반별 평균 모의고사 성적 향상률, 리포트 오류 빈도

핵심 Pain: 여러 시스템 오가며 복사-붙여넣기, 엑셀 실수, 업무 과부하

디지털 리터러시: 중 (엑셀 중급, 단계별 가이드 있으면 빠르게 적응)

페르소나 3: 이관리 (28세, 운영 관리 담당자)

역할: 운영 관리 직원 (학생 수 약 200명), 리포트 원천 데이터 입력 및 관리

목표: 데이터 입력 오류 최소화, 반복 작업 자동화, 시스템 간 데이터 동기화

KPI: 데이터 입력 정확도, 데이터 입력 소요 시간, 시스템 간 데이터 불일치 발생 빈도

핵심 Pain: 여러 시스템 간 데이터 통합 어려움, 수동 입력 실수, 검증 메커니즘 부재

디지털 리터러시: 중하 (기본 엑셀 사용 가능, 클릭 기반 메뉴 방식 인터페이스 선호, 단계별 가이드 제공 시 빠르게 적응)

4.2 사용자 권한 및 역할

원장/관리자: 모든 기능 접근, 리포트 생성, 시스템 설정, 사용자 관리

학사 관리자: 리포트 생성, 학생 데이터 조회, 위험 신호 확인, 리포트 전송 설정

운영 관리자: 데이터 입력, 시스템 연동 관리, 리포트 원천 데이터 관리

### 4.3 AI 자동화 범위 vs 사용자 수동 작업 범위

본 제품에서 **AI/시스템이 자동으로 수행**하는 작업과 **사용자가 수동으로 수행**해야 하는 작업을 명확히 구분한다.

#### 시스템 자동화 범위 (AI/시스템이 자동 수행)
1. **데이터 수집 및 통합**
   - API 연동을 통한 외부 시스템 데이터 자동 수집 (출석 앱, LMS, 모의고사 플랫폼)
   - 파일 업로드 시 데이터 자동 파싱 및 저장
   - 시스템 간 데이터 불일치 자동 감지 및 알림

2. **리포트 생성**
   - 학생별 종합 성과 리포트 자동 생성 (버튼 클릭 시 30초 이내)
   - 리포트 템플릿 자동 적용 및 레이아웃 생성
   - 그래프 및 차트 자동 생성

3. **인사이트 생성**
   - 학습 시간 감소, 출석률 하락, 성적 하락 등 위험 신호 자동 감지
   - 위험도 점수 자동 계산 (0~100점)
   - 자동 인사이트 텍스트 생성 (예: "최근 2주간 학습 시간이 목표 대비 20% 감소했습니다")

4. **스케줄 작업**
   - 정기 리포트 자동 생성 및 전송 (설정된 스케줄에 따라)
   - 위험 신호 자동 분석 (매일 오전 9시)
   - 데이터 자동 동기화 (API 연동 시 설정된 주기에 따라)

5. **알림 전송**
   - 위험 신호 감지 시 자동 알림 전송 (이메일/SMS)
   - 리포트 전송 완료 알림
   - 데이터 불일치 감지 시 자동 알림

#### 사용자 수동 작업 범위 (사용자가 직접 수행)
1. **초기 설정**
   - 시스템 연동 설정 (API 키 입력, 파일 업로드 방식 선택)
   - 리포트 템플릿 설정 (학원 로고 업로드, 색상 선택)
   - 알림 수신자 설정 (관리자, 학사 관리자 이메일/SMS 번호)
   - 위험 신호 감지 기준 설정 (출석률 하락 임계값, 학습 시간 감소 임계값 등)

2. **데이터 입력 (API/파일 업로드 불가능한 경우)**
   - 수동 입력 인터페이스를 통한 데이터 입력
   - 시스템 간 데이터 불일치 수정 (불일치 목록 확인 후 수동 수정)

3. **리포트 생성 요청**
   - 학생 검색 후 "리포트 생성" 버튼 클릭 (F1)
   - 리포트 생성 후 다운로드 및 학부모 전달

4. **리포트 검토 및 전달**
   - 생성된 리포트 내용 검토 (자동 생성이지만 사용자가 최종 확인)
   - 리포트를 학부모에게 전달 (이메일 첨부, 인쇄 등)

5. **위험 신호 대응**
   - 위험 신호 감지 후 학생 관리 조치 (상담, 컨택 등)
   - 조치 완료 후 시스템에 조치 내역 기록 (선택 사항)

**핵심 원칙**: 시스템은 데이터 수집, 리포트 생성, 위험 감지, 알림 전송 등 **반복적이고 규칙적인 작업을 자동화**하며, 사용자는 **초기 설정, 최종 검토, 개입이 필요한 조치**를 담당한다.

#### AI/시스템 제약사항 및 안전 장치

본 제품의 AI/시스템 자동화 기능은 다음 제약사항과 안전 장치를 준수해야 한다.

**1. AI/시스템이 수행해도 되는 작업 (허용 범위)**
- 데이터 수집 및 통합: API 연동, 파일 파싱, 데이터 검증 및 저장
- 리포트 생성: 템플릿 기반 리포트 자동 생성, 그래프/차트 생성, PDF/PNG 변환
- 인사이트 생성: 규칙 기반 위험 신호 감지, 위험도 점수 계산, 사전 정의된 인사이트 텍스트 생성
- 스케줄 작업: 설정된 스케줄에 따른 자동 실행 (리포트 생성, 위험 신호 분석, 데이터 동기화)
- 알림 전송: 위험 신호 감지 시 사전 정의된 알림 템플릿 기반 전송

**2. AI/시스템이 수행하면 안 되는 작업 (차단 범위)**
- 사용자 승인 없이 데이터 수정/삭제: 시스템은 데이터를 읽고 저장만 하며, 사용자 명시적 승인 없이 기존 데이터를 수정하거나 삭제하지 않음
- 정의되지 않은 기능 실행: PRD에 명시되지 않은 기능은 실행하지 않음
- 무한 루프 또는 무제한 리소스 소모: 재시도 횟수 제한(최대 3회), 실행 시간 제한(리포트 생성 30초, 데이터 동기화 2분), 동시 실행 제한(리포트 생성 최대 10건 동시 처리)
- 사용자 개입이 필요한 의사결정: 위험 신호 감지 후 조치 결정, 리포트 내용 최종 검토, 데이터 불일치 수정 등은 사용자가 직접 수행
- 외부 시스템 직접 제어: 외부 시스템(출석 앱, LMS 등)의 데이터를 읽기만 하며, 외부 시스템 설정 변경이나 데이터 수정은 수행하지 않음

**3. 무한 루프 및 리소스 소모 방지 조건**
- **재시도 제한**: 모든 자동 작업(API 연동, 리포트 생성, 알림 전송)은 최대 3회까지만 재시도하며, 재시도 간격은 최소 5분 이상
- **실행 시간 제한**: 
  - 리포트 생성: 30초 초과 시 실패 처리
  - 데이터 동기화: 시스템당 2분 초과 시 실패 처리
  - 위험 신호 분석: 학생 300명 기준 5분 초과 시 실패 처리
- **동시 실행 제한**: 
  - 리포트 생성: 최대 10건 동시 처리, 초과 시 대기열에 추가
  - 데이터 동기화: 시스템당 순차 처리, 동시 다중 시스템 동기화 불가
- **데이터 크기 제한**: 
  - 파일 업로드: 최대 50MB
  - 리포트 생성 대상: 한 번에 최대 100명까지
  - 위험 신호 분석: 한 번에 최대 500명까지
- **스케줄 실행 제한**: 
  - 정기 리포트 자동 전송: 최소 1시간 간격
  - 위험 신호 분석: 최소 1시간 간격
  - 데이터 동기화: 최소 1시간 간격

**4. 입출력 구조 명확성 원칙**
- 모든 기능(F1~F4)은 **입력(Input)**, **처리(Process)**, **출력(Output)** 구조로 명확히 정의됨
- 입력 데이터의 형식, 필수/선택 여부, 검증 규칙을 명시
- 출력 데이터의 형식, 포함 정보, 저장 위치를 명시
- 정의되지 않은 입력은 처리하지 않으며, 정의되지 않은 출력은 생성하지 않음

**5. 기능 스코프 제한 원칙**
- **정의되지 않은 기능은 포함하지 않는다**: PRD 6장에 명시된 기능(F1~F4)만 MVP 범위에 포함
- **Post-MVP 기능 차단**: F5~F9는 MVP 개발 중 요청되더라도 별도 승인 없이 추가하지 않음
- **기능 확장 제한**: 각 기능의 상세 요구사항에 명시된 범위를 초과하여 기능을 확장하지 않음

### 4.4 사용자 행동 → 데이터 저장 → 조회 연결 (데이터 플로우)

본 제품에서 **사용자가 수행하는 행동이 어떤 데이터로 저장되고, 어디에서 조회되는지** 명확히 정의한다.

#### F1: 학생별 종합 성과 리포트 즉시 생성

**사용자 행동**:
- 학생 검색 (이름 또는 학생 ID 입력)
- "리포트 생성" 버튼 클릭

**데이터 저장**:
- 리포트 생성 이벤트 로그 저장 (이벤트 타임스탬프, 사용자 ID, 학생 ID, 리포트 생성 시간)
- 리포트 파일 저장 (PDF/PNG 형태, 리포트 ID, 학생 ID, 생성 시간)
- 리포트 생성 이력 저장 (리포트 ID, 학생 ID, 생성 시간, 리포트 유형, 다운로드 여부)

**데이터 조회**:
- 학생 데이터 조회: `students` 테이블 (학생 기본 정보) → `attendance` 테이블 (출석 데이터) → `study_time` 테이블 (학습 시간 데이터) → `mock_exam` 테이블 (모의고사 성적) → `assignments` 테이블 (과제 완료도)
- 리포트 생성 이력 조회: 리포트 생성 이력 페이지에서 리포트 ID, 생성 시간, 다운로드 여부 확인

#### F2: 다중 시스템 데이터 자동 통합

**사용자 행동**:
- 시스템 연동 설정 (API 키 입력 또는 파일 업로드)
- 데이터 통합 대시보드 접근

**데이터 저장**:
- 외부 시스템 데이터 저장: API 연동 시 자동 수집된 데이터 → `attendance` 테이블, `study_time` 테이블, `mock_exam` 테이블, `payment` 테이블에 저장
- 파일 업로드 데이터 저장: 업로드된 CSV/Excel 파일 파싱 → 위 테이블에 저장
- 데이터 동기화 이력 저장: 동기화 시간, 시스템 종류, 동기화 성공/실패 여부

**데이터 조회**:
- 통합 대시보드에서 조회: `attendance` 테이블 (출석률 일별/주별/월별 집계) → `study_time` 테이블 (학습 시간 일별/주별/월별 집계) → `mock_exam` 테이블 (최신 성적, 추이 그래프) → `payment` 테이블 (미납 학생 목록)
- 데이터 불일치 조회: 데이터 불일치 감지 시 알림 및 불일치 목록 표시

#### F3: 학부모 정기 리포트 자동 전송

**사용자 행동**:
- 리포트 전송 설정 (전송 주기, 전송 시간, 전송 대상 선택)
- 리포트 전송 내역 확인

**데이터 저장**:
- 리포트 전송 스케줄 저장: `report_schedule` 테이블 (전송 주기, 전송 시간, 전송 대상, 활성화 여부)
- 리포트 전송 이력 저장: `report_delivery` 테이블 (리포트 ID, 학생 ID, 학부모 이메일, 전송 시간, 전송 성공/실패 여부, 열람 여부, 열람 시간)
- 리포트 생성 및 전송 이벤트 로그 저장

**데이터 조회**:
- 리포트 전송 내역 조회: 리포트 전송 내역 페이지에서 리포트 ID, 전송 시간, 열람 여부, 열람 시간 확인
- 리포트 열람 여부 추적: 리포트 링크 클릭 시 열람 시간 기록 및 조회

#### F4: 학생별 위험 신호 자동 감지 및 알림

**사용자 행동**:
- 위험 신호 감지 기준 설정 (출석률 하락 임계값, 학습 시간 감소 임계값 등)
- 주의 필요 학생 리스트 확인
- 위험 신호 감지 후 조치 기록 (선택 사항)

**데이터 저장**:
- 위험 신호 감지 결과 저장: `risk_detection` 테이블 (학생 ID, 감지 시간, 위험 신호 종류, 위험도 점수, 감지 기준)
- 알림 전송 이력 저장: `risk_alert` 테이블 (학생 ID, 알림 전송 시간, 알림 수신자, 알림 내용)
- 조치 기록 저장: `risk_action` 테이블 (학생 ID, 위험 신호 ID, 조치 시간, 조치 내용, 조치자) - 사용자가 선택적으로 기록

**데이터 조회**:
- 주의 필요 학생 리스트 조회: 위험도 점수 기준 내림차순 정렬하여 조회, 학생별 위험 신호 상세 정보 표시
- 위험 신호 이력 조회: 학생별 위험 신호 감지 이력, 알림 전송 이력, 조치 기록 확인

**데이터베이스 스키마 요약**:
- `students`: 학생 기본 정보 (학생 ID, 이름, 반, 지점 등)
- `attendance`: 출석 데이터 (학생 ID, 날짜, 출석 여부 등)
- `study_time`: 학습 시간 데이터 (학생 ID, 날짜, 학습 시간 등)
- `mock_exam`: 모의고사 성적 (학생 ID, 모의고사 회차, 성적 등)
- `assignments`: 과제 완료도 (학생 ID, 과제 ID, 완료 여부 등)
- `payment`: 결제 현황 (학생 ID, 납부 여부 등)
- `report_schedule`: 리포트 전송 스케줄
- `report_delivery`: 리포트 전송 이력
- `risk_detection`: 위험 신호 감지 결과
- `risk_alert`: 위험 신호 알림 이력
- `risk_action`: 위험 신호 조치 기록

### 6-1. MVP 기능 (1차 릴리즈)

#### F1. 학생별 종합 성과 리포트 즉시 생성

**우선순위**: High  
**MVP 포함**: Yes

**⚠️ 기능 스코프 제한: 본 기능은 아래에 명시된 입력, 처리, 출력 구조만 포함하며, 정의되지 않은 기능은 포함하지 않는다.**

**입력 (Input)**:
- 사용자 입력: 학생 이름 또는 학생 ID (검색 필드 입력)
- 사용자 액션: "리포트 생성" 버튼 클릭
- 시스템 데이터: 
  - 출석 데이터 (`attendance` 테이블: 학생 ID, 날짜, 출석 여부)
  - 학습 시간 데이터 (`study_time` 테이블: 학생 ID, 날짜, 학습 시간)
  - 모의고사 성적 (`mock_exam` 테이블: 학생 ID, 모의고사 회차, 성적)
  - 과제 완료도 (`assignments` 테이블: 학생 ID, 과제 ID, 완료 여부)
  - 반 평균 데이터 (같은 반 학생들의 평균 데이터)

**처리 (Process)**:
1. 학생 데이터 조회: 학생 ID로 관련 테이블에서 데이터 조회
2. 데이터 집계:

   - 전송 내역 기록: 전송 시간, 전송 대상, 전송 성공/실패 여부
   - 리포트 열람 여부 추적: 리포트 링크 클릭 시 열람 시간 기록
   - 전송 실패 시 재전송 기능 제공

6. **템플릿 커스터마이징 (기본)**
   - 학원 로고 업로드 및 리포트에 자동 포함
   - 기본 색상 설정 (최대 3가지 색상)
   - 리포트 레이아웃 선택 (2가지 기본 템플릿)

7. **에러 처리**
   - 이메일 전송 실패 시: 재시도 로직 (최대 3회, 1시간 간격)
   - 리포트 생성 실패 시: 해당 학생 제외하고 나머지 학생 전송 후 관리자에게 알림

**해결하는 Pain**:
- 정기 리포트 제작 및 전송에 주 10~20시간 소비 (JTBD #3)
- 리포트 형식 불일치로 인한 전문성 저하
- 리포트 전송 누락 시 학부모 불만 발생

**기대 효과**:
- 리포트 전송 업무 시간 주 15시간 → 0시간 (완전 자동화)
- 일관된 리포트로 학원 브랜드 이미지 향상

#### F4. 학생별 위험 신호 자동 감지 및 알림

**우선순위**: High  
**MVP 포함**: Yes

**⚠️ 기능 스코프 제한: 본 기능은 아래에 명시된 입력, 처리, 출력 구조만 포함하며, 정의되지 않은 기능은 포함하지 않는다.**

**입력 (Input)**:
- 사용자 입력: 위험 신호 감지 기준 설정 (출석률 하락 임계값: 20% 이상 하락 또는 주간 출석률 70% 이하, 학습 시간 감소 임계값: 30% 이상 감소 또는 주간 학습 시간 목표 대비 50% 미만, 모의고사 성적 하락 임계값: 최근 2회 평균 대비 10점 이상 하락, 과제 완료도 임계값: 주간 과제 완료율 50% 미만)
- 시스템 자동 실행: 매일 오전 9시에 자동으로 모든 학생 데이터 분석
- 시스템 데이터:
  - 학생 데이터 (출석률, 학습 시간, 모의고사 성적, 과제 완료도)
  - 전주 대비 데이터 (비교 기준)

**처리 (Process)**:
1. 위험 신호 감지 기준 확인: 설정된 임계값 확인
2. 학생별 데이터 분석:
   - 출석률 분석: 전주 대비 20% 이상 하락 또는 주간 출석률 70% 이하 확인
   - 학습 시간 분석: 전주 대비 30% 이상 감소 또는 주간 학습 시간 목표 대비 50% 미만 확인
   - 모의고사 성적 분석: 최근 2회 평균 대비 10점 이상 하락 확인
   - 과제 완료도 분석: 주간 과제 완료율 50% 미만 확인
3. 위험도 점수 계산 (0~100점):
   - 출석률 하락: 30점
   - 학습 시간 감소: 25점
   - 성적 하락: 25점
   - 과제 완료도 저하: 20점
   - 여러 위험 신호가 동시에 발생 시 점수 누적 (최대 100점)
4. 위험 신호 감지 결과 저장:
   - 위험 신호 감지 결과 저장 (`risk_detection` 테이블: 학생 ID, 감지 시간, 위험 신호 종류, 위험도 점수)
5. 알림 전송:
   - 위험 신호 감지 시 즉시 알림 전송 (이메일/SMS, 관리자/학사 관리자에게)
   - 알림 내용: 학생 이름, 위험 신호 종류, 위험도 점수, 상세 정보 링크
   - 알림 전송 이력 저장 (`risk_alert` 테이블)

**출력 (Output)**:
- 주의 필요 학생 리스트:
  - 위험도 점수 기준 내림차순 정렬
  - 학생별 위험 신호 상세 정보 표시:
    - 어떤 지표에서 위험 신호가 감지되었는지 (출석률, 학습 시간, 성적 등)
    - 전주 대비 변화율
    - 위험도 점수 및 우선순위
  - 리스트는 최대 50명까지 표시 (페이지네이션)
- 알림:
  - 위험 신호 감지 시 즉시 알림 전송 (이메일/SMS)
  - 알림 수신자: 관리자, 학사 관리자 (설정 가능)
- 데이터 저장:
  - 위험 신호 감지 결과 저장 (`risk_detection` 테이블)
  - 알림 전송 이력 저장 (`risk_alert` 테이블)
  - 조치 기록 저장 (`risk_action` 테이블, 사용자가 선택적으로 기록)
- 성능 기준: 분석 완료 시간 학생 300명 기준 5분 이내, 알림 전송 시간 1분 이내

**상세 요구사항**:
1. **위험 신호 감지 기준 (설정 가능)**
   - 출석률: 전주 대비 20% 이상 하락 또는 주간 출석률 70% 이하
   - 학습 시간: 전주 대비 30% 이상 감소 또는 주간 학습 시간 목표 대비 50% 미만
   - 모의고사 성적: 최근 2회 평균 대비 10점 이상 하락
   - 과제 완료도: 주간 과제 완료율 50% 미만

2. **위험도 점수 계산**
   - 점수 범위: 0~100점 (점수 높을수록 위험)
   - 계산 방식:
     - 출석률 하락: 30점
     - 학습 시간 감소: 25점
     - 성적 하락: 25점
     - 과제 완료도 저하: 20점
   - 여러 위험 신호가 동시에 발생 시 점수 누적 (최대 100점)

3. **주의 필요 학생 리스트**
   - 위험도 점수 기준 내림차순 정렬
   - 학생별 위험 신호 상세 정보 표시:
     - 어떤 지표에서 위험 신호가 감지되었는지
     - 전주 대비 변화율
     - 위험도 점수 및 우선순위
   - 리스트는 최대 50명까지 표시 (페이지네이션)

4. **자동 알림**
   - 위험 신호 감지 시 즉시 알림 전송 (이메일/SMS)
   - 알림 수신자: 관리자, 학사 관리자 (설정 가능)
   - 알림 내용: 학생 이름, 위험 신호 종류, 위험도 점수, 상세 정보 링크
   - 알림 전송 주기: 실시간 (위험 신호 감지 즉시)

5. **알림 분석 주기**
   - 매일 오전 9시에 자동으로 모든 학생 데이터 분석
   - 분석 완료 시간: 학생 300명 기준 5분 이내

6. **조치 추적**
   - 위험 신호 감지 후 관리자 조치 여부 추적
   - 조치 미완료 시 24시간 후 재알림
   - 조치 완료 시 위험 신호 해제 및 이력 기록

7. **에러 처리**
   - 데이터 부재 시: 해당 학생 제외하고 분석 진행
   - 알림 전송 실패 시: 재시도 로직 (최대 3회, 1시간 간격)

**해결하는 Pain**:
- 학생별 데이터 수동 확인의 비효율 (JTBD #5)
- 문제 학생 조기 발견의 어려움
- 우선순위 설정의 애매함

**기대 효과**:
- 문제 학생 조기 발견으로 개입 시점 선제적 대응 가능
- 학생 이탈률 감소 및 재등록률 향상
